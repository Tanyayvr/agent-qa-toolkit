Agent QA Toolkit
Self-hosted слой qualification и evidence для AI-агентов
- Portable evidence packs
- Runtime-classed validation paths
- Детерминированные CI/release gates
- ЦА: engineering leads, platform teams, QA, AI infrastructure

Проблема
- Команды видят trace, но всё равно не могут доказать, можно ли доверять агенту как части release workflow
- "Зелёный" прогон часто смешивает runtime failure, timeout budget и model quality в один шумный вердикт
- Тяжёлых локальных агентов гоняют тем же циклом, что и быстрых, теряя часы и не понимая реальную причину
- Support/on-call по-прежнему восстанавливают контекст по скриншотам и кускам логов

Что мы узнали на реальных прогонах
- не каждый агент подходит под один и тот же локальный validation loop
- маленький репозиторий не означает маленький runtime
- боль с timeout часто говорит не только о конфиге, а об operational fitness агента
- командам нужно понимать, какой путь валидации для этого агента вообще честный

Тезис продукта
Мы не просто тестируем агентов.
Мы определяем, operationally fit ли агент для реального engineering/release цикла.

Ключевое разделение
- execution quality: был ли runtime вообще пригоден для оценки?
- model quality: был ли output/behavior приемлемым?
- operational fitness: какой validation path этот агент реально поддерживает?

Validation paths
- quick: readiness / triage
- full-lite: практический локальный regression path
- full: полная quality certification
- diagnostic: длинный investigation path для slow but live агентов
- nightly / dedicated runs: честный путь для тяжёлых агентов

Runtime classes
- fast_remote
- standard_cli
- slow_local_cli
- heavy_mcp_agent

Почему это важно
- оператор перестаёт считать, что все агенты обязаны жить в одном и том же loop
- runtime truth становится частью quality truth
- toolkit говорит не только "что сломалось", но и "какой путь валидации нужен дальше"

Что даёт артефакт
- report.html для человека
- compare-report.json для CI и автоматизации
- stage-result.json с typed stage/reason/next_action
- devops-envelope.json с точным runtime envelope
- next-envelope.json с рекомендованным следующим mode и budget

Что мы уменьшаем
- часы на реконструкцию контекста
- ложные выводы о model regression, когда execution был degraded
- бессмысленные full-прогоны, если quick уже показал, что интеграция не готова
- ручной подбор timeout маленькими шагами

Где мы выигрываем
- self-hosted evidence вместо зависимости от dashboard
- qualification layer поверх eval и observability
- runtime class + right-sized validation path
- воспроизводимые release artifacts для support, vendor review и governance

Promise для инди
- quick + full-lite за минуты, если агент это поддерживает
- локальный артефакт без enterprise-overload

Promise для enterprise
- quick + full + diagnostic + portable evidence + deterministic gates
- reproducibility и governance для release decisions

CTA

- Сначала запустите quick
- Посмотрите, готов ли агент к full, подходит ли только для full-lite, или честнее сразу идти в diagnostic/nightly path
