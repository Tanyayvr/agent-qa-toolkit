Agent QA Toolkit
Self-hosted qualification and evidence layer for AI agents
- Portable evidence packs
- Runtime-classed validation paths
- Deterministic CI/release gates
- Target: engineering leads, platform teams, QA, AI infrastructure

Problem
- Teams can see traces, but still cannot prove whether an agent is safe to trust in a release workflow
- "Green" often mixes runtime failure, timeout budget, and model quality into one noisy verdict
- Heavy local agents are run through the same loop as fast agents, which wastes hours and hides the real issue
- Support/on-call still reconstruct failures from screenshots and partial logs

What we learned from real agent runs
- not every agent fits the same local validation loop
- small repo does not mean small runtime
- timeout pain is often a signal about operational fitness, not only configuration
- teams need to know which path the agent honestly supports

Product thesis
We do not just test agents.
We determine whether an agent is operationally fit for a real engineering or release loop.

Core split
- execution quality: was the runtime trustworthy?
- model quality: was the behavior/output acceptable?
- operational fitness: what validation path does this agent support?

Validation paths
- quick: readiness / triage
- full-lite: practical local regression path
- full: full quality certification
- diagnostic: long-run investigation for slow but live agents
- nightly / dedicated runs: honest path for heavy agents

Runtime classes
- fast_remote
- standard_cli
- slow_local_cli
- heavy_mcp_agent

Why this matters
- operators stop treating all agents as if they belong in the same loop
- runtime truth becomes part of quality truth
- the toolkit tells you not only what failed, but what path should be used next

What the artifact gives you
- report.html for humans
- compare-report.json for CI and automation
- stage-result.json with typed stage/reason/next_action
- devops-envelope.json with the exact runtime envelope used
- next-envelope.json with the recommended next mode and budget

What we reduce
- hours spent reconstructing context
- false model-regression conclusions when execution was degraded
- pointless full runs when quick already shows the integration is not ready
- manual timeout tuning by tiny increments

Where we win
- self-hosted evidence, not dashboard dependency
- qualification layer above eval and observability
- runtime class + right-sized validation path
- reproducible release evidence for support, vendor review, and governance

Indie promise
- quick + full-lite in minutes when the agent supports it
- local artifact, no enterprise overhead

Enterprise promise
- quick + full + diagnostic + portable evidence + deterministic gates
- reproducibility and governance for release decisions

Call to action
- Connect a real agent
- Run quick first
- See whether the agent is ready for full, only fit for full-lite, or belongs on a diagnostic/nightly path
