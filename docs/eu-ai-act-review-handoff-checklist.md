# EU AI Act Review Handoff Checklist

Use this checklist when handing the evidence bundle to:

- internal audit
- compliance review
- release approval board
- external counsel
- customer-side technical reviewers

## 1. Package Scope Statement

State this explicitly in the handoff note:

- this package is technical evidence, not legal advice
- it covers runtime validation and evidence exports produced by the evaluator
- legal classification, deployer instructions, and formal declarations remain outside this package

## 2. Minimum Files To Attach

Always attach:

- `compare-report.json`
- `report.html`
- `artifacts/manifest.json`
- `review/review-decision.json`
- `review/handoff-note.md`
- `compliance/eu-ai-act-coverage.json`
- `compliance/eu-ai-act-annex-iv.json`
- `compliance/eu-ai-act-report.html`
- `compliance/evidence-index.json`
- `compliance/article-13-instructions.json`
- `compliance/article-9-risk-register.json`
- `compliance/article-17-qms-lite.json`
- `compliance/article-72-monitoring-plan.json`
- `compliance/human-oversight-summary.json`
- `compliance/release-review.json`
- `compliance/post-market-monitoring.json`

If the review is release-critical, also attach:

- relevant `case-*.html` diffs
- `assets/raw/run_meta/*`
- grouped bundle index if multiple runs were reviewed together

## 3. Integrity Checks

Confirm before sending:

- `quality_flags.self_contained = true`
- `quality_flags.portable_paths = true`
- `missing_assets_count = 0`
- `path_violations_count = 0`

If any of these fail, say so plainly in the handoff note.

Also confirm:

- `npm run review:check -- --reportDir <dir>` exits cleanly
- the handoff note does not contain `TODO`
- every machine-derived residual gap in `review/review-decision.json` has an owner and disposition
- `review/review-decision.json.human_completion.eu_scaffold_completion` is completed for Article 13, Article 17, and Article 72
- each required input inside those EU scaffold sections is marked `completed` or `deferred` with a note
- residual gaps in those EU scaffold sections are explicitly acknowledged by a named owner

## 4. Governance Checks

Call out these values:

- execution quality status
- release decision status
- cases requiring approval
- cases recommended for block
- monitoring status and whether current run is included in history
- whether the Article 13 instructions scaffold, Article 9 risk register, Article 17 QMS-lite scaffold, and Article 72 monitoring plan still need owner-authored completion
- whether `review/review-decision.json` records who completed the Article 13, Article 17, and Article 72 owner loop
- residual gaps from `compliance_coverage` or Annex IV export

## 5. Sample Handoff Note

Suggested note:

> Attached is the EU AI Act technical evidence bundle for report `<report_id>`. This package contains clause coverage, Annex IV-oriented export, Article 13 instructions scaffold, Article 9 risk register scaffold, Article 17 QMS-lite scaffold, Article 72 monitoring-plan scaffold, human oversight queue, release review, and post-market monitoring outputs. It is a technical evidence package only and does not replace legal classification or legal sign-off.

The structured review scaffold is generated with:

```bash
npm run review:init -- --reportDir <dir> --profile <name>
npm run review:check -- --reportDir <dir> --profile <name>
```

## 6. Questions Reviewers Should Be Able To Answer

The package should let reviewers answer:

- What changed in this release or run?
- Are any cases blocked or approval-required?
- Is execution quality healthy or degraded?
- What evidence supports traceability and record-keeping?
- What residual gaps remain outside evaluator scope?
- Does historical monitoring show drift or worsening gate behavior?

## 7. If The Package Is For Counsel

Add context that counsel usually still needs:

- intended purpose statement
- system boundary description
- deployer/operator context
- ownership of declaration and policy approvals

Those are not generated by the evaluator and should be added by the operator.

## 8. If The Package Is For Internal Audit

Highlight:

- manifest-backed integrity
- machine-readable clause coverage
- explicit residual gaps
- release governance outputs
- monitoring history window and drift summary

## 9. If The Package Is For Release Approval

Focus the handoff on:

- `release-review.json`
- `human-oversight-summary.json`
- selected `case-*.html` diffs
- `post-market-monitoring.json`

The approver should not need to read every raw artifact unless escalation is required.

## 10. Explicit Non-Claims

Do not write:

- "this proves compliance"
- "this replaces legal review"
- "this is the full Annex IV technical file"

Do write:

- "this package supplies technical runtime evidence"
- "this package supports internal review and conformity preparation"
- "this package documents residual gaps explicitly"

## 11. Final Send/No-Send Check

Before sending, confirm:

- the right report directory was selected
- the compliance profile used was the intended one
- the trend DB window is current enough for monitoring review
- the reviewer audience is named in the handoff note
- the scope boundary statement is included
